# mizunomi 構想メモ：「飲む」の中立レイヤー戦略

- 作成日: 2026-07-20
- 位置づけ: 長期構想のメモ。現行MVPの仕様・実装方針（README / 各Issue）には影響しない。
- 出発点: 「記録したら広告収益で途上国に井戸を掘る（UNICEF等と連携）」というアイデアの検討から。

---

## 核となる考え方：行為の抽象度がそのまま市場の広さになる

乗換案内（NAVITIME / Yahoo!乗換案内 / Google Maps）が電鉄各社の公式アプリに勝ったのは、
**人の移動が1社の路線で完結しない**からである。ユーザーの課題は「A地点からB地点」であって
「特定の鉄道会社に乗ること」ではない。だから中立な横断レイヤーに需要が生まれた。

同じ構造を「動詞」で整理すると：

```text
移動する     → Google Maps / NAVITIME
食べる       → あすけん / クックパッド
寝る         → 睡眠トラッカー各種
歩く・運動   → Google Fit / 万歩計系
飲む         → 空白
```

**「飲む」はまだ中立レイヤーが存在しない空白地帯**である。

---

## 飲料版のアナロジー対応表

```text
電鉄各社の公式アプリ        ↔ 飲料メーカーの公式アプリ（Coke ON など）
移動の全行程                ↔ その人が1日に飲んだ全部
時刻表（公共データ）        ↔ 家庭内飲用データ（※どこにも存在しない）
中立な乗換案内              ↔ ブランド横断の「飲む」記録レイヤー
```

重要な非対称性：

- Coke ON（4,000万DL超）が見えるのは「自販機で自社製品を買った瞬間」だけ
- 実際の水分補給の大半は **水道水・家で淹れたお茶・ウォーターサーバー** であり、
  どのメーカーのアプリにも一生記録されない
- 「その人が飲んだ100%」を見られるのは中立アプリだけ

さらに、「家庭内で何を・いつ・どれだけ飲んでいるか」のデータは
POS（買った瞬間）にも自販機アプリにも存在しない。
**中立な記録レイヤーだけがこのデータセットを作れる**。これ自体がメーカーの参加動機になり得る。

---

## 「飲む」の抽象度マップ

日本語の「飲む」は「薬を飲む」まで含む（英語では drink / take に分かれる）。
つまり「飲むログ」の部分集合として以下が同じ記録行為の上に乗る：

```text
水分補給       → 現在のmizunomi（入口）
カフェイン管理 → コーヒー・お茶（すでに記録可能）
アルコール管理 → すでにカテゴリと注意表示がある
服薬・サプリ   → 将来の拡張。製薬・ドラッグストア・保険という別業界がパートナー候補
```

現在のmizunomiは、アルコールまで記録できる時点で、
すでに「水アプリ」ではなく「飲むアプリ」として設計されている。

**自制点**: 抽象度の高さは「拡張の地図」であって「v1の機能一覧」ではない。
入口は「水分習慣」のまま尖らせる。NAVITIMEも乗換案内一点突破から始まった。

## 実証済みの前例

```text
Volvic「1L for 10L」        1L購入→マリに10Lの水（日本ユニセフ協会と提携、2007〜2016）
UNICEF Tap Project          スマホを触らない時間に応じてスポンサーが水支援に寄付
Ecosia                      検索広告収益で植林
Charity Miles               走った距離に応じてスポンサーが寄付
あすけん                    食品版の中立記録レイヤー。中立にユーザーを集め、
                            その上でメーカータイアップ・広告・データ事業を成立させた
```

あすけんは「メーカーの枠を超えた記録レイヤーが、後からメーカーをパートナーとして迎える」
順序の日本国内での実証例。水分版のこのポジションはまだ空いている。

## 日本市場の追い風

### 無糖優位という特異市場

業界統計の「清涼飲料水」はお茶・水を含む大分類であり、本質的な対立軸は加糖 vs 無糖。
日本は加糖優位の欧米と逆で、**無糖飲料（お茶・水・無糖コーヒー）が市場の過半**を占め、
現在も伸び続けている。

```text
カテゴリ最大: 茶系飲料（お〜いお茶は世界で最も売れている緑茶飲料）
ブランドNo.1: サントリー天然水（近年、国内販売数量トップ）
最成長カテゴリ: ミネラルウォーター
```

戦略的な意味：mizunomiの注意表示は「次は水かお茶にしませんか」と言うアプリである。
加糖優位の市場ならこの助言は業界と敵対するが、
**日本ではアプリの助言が市場トップセラーの商業的利益とそのまま一致する**。
水・お茶を勧める中立アプリは、伊藤園・サントリー等にとって
カテゴリスポンサーとして参加する理由しかない。

### 夏 = 熱中症という国民的文脈

```text
環境省・気象庁の熱中症警戒アラートが出るたびに
「こまめな水分補給を」と全国に呼びかけられる
→ 国が毎年、水分記録アプリの需要を喚起してくれる構造
```

- 自治体の健康事業・学校・建設現場など、B2B/公共の入口もここにある
- ポカリ・OS-1（経口補水液）など熱中症文脈を主戦場とするメーカーも当事者
- 将来機能の候補: 「熱中症モード」（警戒アラート日は目標量とリマインド頻度を調整）

### 季節戦略

夏の需要スパイクは毎年必ず来る。Google Play公開には
開発者アカウント＋クローズドテスト（個人アカウントはテスター12人×14日間）の
リードタイムがあるため、無理に当年夏へ突っ込むより、
足場（品質・テスト・Play公開準備）を固めて**翌夏の梅雨明けに照準を合わせる**方が堅実。
熱中症警戒アラートの発令日がそのままグロースのスパイク日になる設計は毎年使える。

## 共同基金モデル：中立だからこそUNICEF案が業界横断になる

単独メーカーのキャンペーン（Volvic型）の構造的限界：

```text
自社製品を買った分しかカウントできない
ライバル製品の消費をカウントする理由がない
キャンペーンが終われば消える
```

中立プラットフォームなら：

```text
どのブランドを飲んでも、水道水でも、1杯 = ◯L の支援
参加メーカーは共同基金に出資（カテゴリスポンサー等の形）
業界横断のCSRインフラとして恒久化できる
```

「乗換案内に各社が時刻表を出さざるを得なかった」のと同様、
**枠組みが中立だからこそ各社が乗れる**。

### 寄付規模の現実

- 井戸1基は支援団体経由で数十万〜200万円程度
- 広告収益はDAU 1,000人で月1.5〜3万円程度が目安（バナー中心の場合）
- よって最初は井戸ではなく **浄水剤・経口補水塩など単価の小さい支援** に紐づける
  - 「今日の記録で20Lの水が浄化できる分の寄付になりました」のように1記録=◯Lで見せる
- 井戸は「みんなの累計」で目指す長期目標として置く

### 信頼性の担保（必須）

```text
寄付証明の公開（領収書等）
累計寄付額のアプリ内表示
「UNICEFとコラボ」を名乗るには事前の契約・審査が必要
  （提携せずに寄付実績を公開する形なら自由。WaterAid等、
   小規模でも組みやすい団体から始める選択肢もある）
```

## パートナーの現実的な順序

最初のパートナーは大手飲料メーカーではない可能性が高い。

```text
第1段階: ブランド非依存で「水を飲む行為そのもの」が増えると儲かる企業
         例: サーモス・象印（マイボトル）、ブリタ（浄水器）、ウォーターサーバー各社
         → 中立性と利害が完全に一致。マイボトル→ペットボトル削減→水支援のSDGs文脈も接続

第2段階: カテゴリスポンサー（麦茶枠、スポーツドリンク枠など）として飲料メーカーを迎える
         → 中立性を保ったまま（独占は売らない、スポンサー表示は明示）

第3段階: 業界横断の共同CSR基金
         → 服薬・保険など「飲む」の他レイヤーへの拡張もこの段階以降
```

## 段階ロードマップ

```text
Phase 1: 単体ユーティリティとしての完成度（現在地）
         Issue 27〜30（バグ修正・リファクタ・テスト・品質）
         通知（Issue #20）→ 設定（Issue #31）の機能開発再開
Phase 2: 広告導入 + 透明な寄付
         リワード広告（「動画を見て1杯分の支援」）+ プライバシーポリシー整備
         収益から定額を寄付し、アプリ内に「みんなの支援」累計ページ
Phase 3: 第1段階パートナー（マイボトル・浄水器・サーバー系）
Phase 4: カテゴリスポンサー → 業界共同基金
```

注意: Phase 2以降は、これまでの「外部ライブラリを追加しない・ネットワーク機能なし」
方針からの大きな転換になる。着手前にリファクタ（Issue 28）が完了していることを前提とする。

## 守るべき原則

```text
中立性を売らない（独占契約をしない。スポンサーはカテゴリ単位・明示表示）
寄付は検証可能にする（できない約束をしない）
医療アプリではないという一線を維持する（README「医療アプリではないこと」に準拠）
記録データはプライバシー最優先で扱う（外部提供は匿名化・統計化が前提）
ユーザーにとっての価値（習慣化支援）が常に先。事業構想のためにUXを曲げない
```
